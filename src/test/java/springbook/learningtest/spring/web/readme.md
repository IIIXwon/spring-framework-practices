## DispatcherServletrhk MVC 아키텍쳐
> * MVC 아키텍쳐는 프론트 컨트롤러 패턴과 함꼐 사용된다, 스프링 MVC에서는 DispatcherServlet이 프론트 컨트롤러를 가르킨다
> * 프론트 컨트롤러는 프리젠테이션 계층의 제일 앞에 둬서 서버로 들어오는 모든 요청을 먼저 받아 공통 작업을 먼저 수행한 후에 적절한 세부 컨트롤러로 작업을 위임해 주고, 클라이언트에게 보낼 뷰를 선택해서 최종 결과를 생성하는 등의 작업을 수행한다, 예외가 발생했을 떄 이를 일관된 방식으로 처리하는 것도 프론트 컨트롤러의 역할이다
## 작업의 흐름
> * Http 요청 접수
> > * 자바 서버의 서블릿 컨테이너는 Http 프로토콜을 통해 들어오는 요청이 DispatcherServlet에 할당된 것이라면 (URL 매칭) http 요청정보를 DispatcherServlet에 전달해준다, 
> * DispatcherServlet에서 컨트롤러로 http 요청 위임
> > * 어떤 컨트롤러에게 작업을 위임 할지 결정 할떄 핸들러 매핑 전략을 이용한다, 스프링에서는 `컨트롤러 = 핸들러`라고 한다, 웹의 요청을 다루는 오브젝트라는 의미다
> > * 사용자 요청을 기준으로 어떤 핸들러에게 작업을 위임할지를 결정해주는 것을 핸들러 매핑 전략이라고 한다
> > * 전략이라고 부르는 이유는 DI의 가장 대표적인 용도라고 할 수 있는 전략 패턴이 적용되어 있기 떄문이다, 어떤 URL이 들어오면 어떤 컨트롤러 오브젝트가 이를 처리하게 할지르 매핑해주는 전략을 만들어 DI로 제공해주기만 하면 된다
> > * 어떤 핸들러가 요청을 처리하게 할지 결정했다면, 다음은 해당 컨트롤러 오브젝트의 메소드를 호출해서 실제로 웹 요청을 처리하는 작업을 위임한다, 하지만 제각각 다른 메소드와 포맷을 가진 오브젝트를 컨트롤러를 DispatcherServlet이 호출하게 하는 방법은 
> > * 오브젝트 어댑터 패턴을 사용해서, 특정 컨트롤러를 호출해야 할 떄는 해당 컨트롤러 타입을 지원하는 어댑터를 중간에 껴서 호출하면 항상 일정한 방식으로 컨트롤러를 호출하고 결과를 받을 수 있다
> * 모델 생성과 정보 등록
> > * MVC 패턴의 장점 : 정보를 담고 있는 모델(M)과 정보를 어떻게 보여줄지 알고 있는 뷰(V)가 분리된다는 점이다
> > * 컨트롤러의 작업은 먼저 사용자 요청을 해석, 실제 비지니스 로직을 수행하도록 서비스 계층에 작업 위임, 결과를 받아서 모델 생성, 어떤 뷰를 사용할지 결정하는 것, 네 가지로 분류할 수 있다
> > * 컨트롤러가 DispatcherServlet에게 돌려줘야 할 두 가지 정보 : 모델, 뷰
> * 컨트롤러의 결과 리턴 : 모델과 뷰
> > * 컨트롤러에서 결과를 리턴 받으면 DispatcherServlet의 viewResolver가 뷰 오브젝트를 생성한다
> * DispatcherServlet의 뷰 호출
> > * 생성된 뷰 오브젝트에게 모델을 전달해주고 클라이언트에게 최종 결과물을 생성해달라고 요청
> > * 기술적으로 최종 결과물은 HttpServletResponse 오브젝트 안에 담긴다
> * Http 요청 반환
> > * 모든 작업을 마쳤으면 DispatcherServlet은 후처리기를 확인 하고, 있다면 후처리기 작업을 진행한 뒤에 HttpServletResponse에 담긴 최종 결과물을 서블릿 컨테이너(Tomcat)에 전달한다
> > * 서블릿 컨테이너는 HttpServletResponse에 담긴 정보를 http 응답으로 만들어 사용자 브라우저나 클라이언트에 전송하고 작업을 종료한다
